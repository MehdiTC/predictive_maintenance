# Features, Labels, and Splits — NASA C-MAPSS FD001 (Loop 3)

This document is the contract for the Loop 3 feature layer. It consumes the validated Parquet
outputs defined in [data_contract.md](data_contract.md) and produces reproducible, model-ready
feature tables plus split and feature manifests. It stops before any modeling: no training,
scaling, feature selection, evaluation, or imputation happens here.

The build boundary is:

```text
validated FD001 trajectories (Loop 2 Parquet)
        -> RUL label generation
        -> asset-level split assignment
        -> leakage-safe feature generation (shared FeatureBuilder)
        -> model-ready Parquet per partition + split & feature manifests
```

## Remaining Useful Life (RUL) labels

For a training asset `i` observed to fail at its final cycle `T_i`, the RUL at cycle `t` is:

```text
RUL(i, t) = T_i − t
```

Because training trajectories are complete run-to-failure records with contiguous cycles `1..T_i`
(enforced by Loop 2 validation), `T_i` is the asset's maximum cycle, and the uncapped RUL steps
down by exactly one each cycle, reaching `0` at failure.

* **Uncapped `rul`** — always generated (`int64`).
* **Capped `rul_capped = min(rul, cap)`** — generated only when a cap is configured; the uncapped
  `rul` is always preserved. The cap models the common assumption that early-life RUL is not
  meaningfully predictable and is clipped to a constant healthy value. Loop 4 compares raw vs
  capped RUL as a hyperparameter.

Label validation (`validate_rul_labels`) rejects negative RUL, any asset whose final-cycle RUL is
not 0, and any trajectory where RUL does not decrease by exactly 1 as cycle increases by 1.

### Official test labels are not fabricated

The truncated test trajectories carry **no per-row RUL labels** — deriving them would require the
unavailable future cycles. Instead the official `RUL_FD001.txt` value for each test engine (the
true RUL at its truncation cycle) is preserved as an **evaluation benchmark only**:

* `test_labels.parquet` has one row per test asset: `asset_id`, `final_cycle`, `rul`.
* Per the dataset readme, official file row `i` corresponds to test `asset_id` `i + 1`. This
  positional correspondence cannot be verified from file content and is asserted explicitly.
* This benchmark is for **evaluation** (predict RUL at each test engine's last observed cycle and
  compare) — never for training or feature-selection decisions.

## Why splits are by asset

Splits are performed by `asset_id`, **never by individual row**. Splitting rows would put cycles
of the same trajectory into both train and validation, leaking near-duplicate, time-adjacent
observations and inflating measured performance. Each asset's entire trajectory lands in exactly
one partition.

| Partition | Fraction | FD001 assets | Role |
| --- | --- | --- | --- |
| `train` | 70 % | 70 | Fit models and any preprocessing |
| `validation` | 15 % | 15 | Model selection and tuning |
| `calibration` | 5 % | 5 | Held out for conformal prediction intervals (Loop 4) |
| `replay` | 10 % | 10 | Held out entirely; streamed as "live" data in Loop 8 |

* **Calibration is separate** from train and validation because Loop 4 uses split conformal
  prediction, which needs a calibration set disjoint from what the model was fit and selected on.
* **Replay is isolated** so held-out engines can be replayed later with delayed failure labels;
  they must never be used for initial training, preprocessing fitting, feature selection,
  calibration, or threshold tuning.
* The official NASA **test** set is untouched and remains a separate final benchmark.

Determinism: the unique training asset IDs are sorted and permuted with
`numpy.random.default_rng(seed)` (a reproducible PCG64 stream), then sliced into contiguous blocks
whose sizes come from the fractions via largest-remainder rounding (counts always sum to the asset
count). The same seed and asset set always yield the same partitions. The seed is configurable and
recorded in the split manifest.

## Feature definitions

Features are generated by the single shared `FeatureBuilder`
(`src/turbine_guard/features/builder.py`), which is the only feature implementation and is reused
by future online inference, monitoring, and retraining — there is no separate notebook or serving
copy.

For each of the 24 source columns (3 operating settings + 21 sensors), the enabled families are:

| Family | Column name | Notes |
| --- | --- | --- |
| Current value | `{col}_current` | The raw value at cycle `t` |
| Previous-cycle difference | `{col}_delta_1` | `x_t − x_{t−1}`; null at the first cycle |
| Rolling mean | `{col}_roll_mean_w{W}` | Trailing window of `W` cycles |
| Rolling std | `{col}_roll_std_w{W}` | Sample std (ddof=1); null with one observation |
| Rolling min / max | `{col}_roll_min_w{W}` / `{col}_roll_max_w{W}` | Trailing window |
| Rolling range | `{col}_roll_range_w{W}` | `max − min` over the window |
| Rolling slope | `{col}_roll_slope_w{W}` | OLS slope of value vs. cycle over the window |
| EWM mean | `{col}_ewm_mean_s{S}` | Exponentially weighted mean, span `S`, `adjust=True` |

Default rolling windows are **5, 10, 20 cycles**; default EWM spans are **5, 10, 20**. With all
families enabled this yields `24 × 23 = 552` feature columns. The enabled families and windows are
configurable; column names are deterministic and the column order is stable (source columns in
canonical order; within each column: current, delta, then each windowed family across ascending
windows, then EWM across ascending spans). `FeatureBuilder.feature_columns()` returns the exact
ordered list, which is also recorded in the feature manifest.

### Rolling-window semantics and early-cycle behaviour

* Windows are **trailing** (right-aligned) and end at the current cycle — no centered or
  forward-looking windows exist.
* All calculations are **grouped by asset** and never cross asset boundaries.
* `min_periods` defaults to **1**, so rolling mean/min/max are defined from the first cycle.
  Families needing at least two points are structurally undefined on the first cycle: `delta_1`
  and `roll_std` are **null**; `roll_slope` is a deterministic **0.0** (a single point, or any
  degenerate fit, has slope 0). Setting `min_periods > 1` makes early windows null instead.
* Rolling slope uses only cycles inside the trailing window and is computed vectorized from grouped
  rolling sums (no per-row Python loop). Its only division is guarded, so no infinities arise.

### Missing-value policy

Early-cycle rows are **preserved, not dropped**. Structurally-unavailable rolling/lag values are
left **null**. Loop 3 **does not impute or scale** — imputation and model-specific scaling are
deferred to the Loop 4 model pipeline, where they can be fit on training assets only and applied
identically to the other splits. The feature manifest records `imputation: null` to make this
deferral explicit, and per-file null counts so the early-cycle nulls are auditable.

## Leakage risks and protections

The central risk is using future observations to build an earlier feature. Because `FeatureBuilder`
holds no fitted state and uses only trailing, asset-grouped windows, leakage safety is structural.
It is proven by explicit tests (`tests/unit/test_feature_leakage.py`):

* **Future-row mutation** — changing sensor values at cycles `> t` leaves every feature at `t`
  unchanged (also verified exhaustively by perturbing each future cycle in turn).
* **Future-row append** — appending later cycles leaves features through `t` unchanged.
* **Cross-asset isolation** — changing one asset's data does not change another asset's features.
* **Fit isolation** — features for a subset are identical with or without other assets present
  (the builder fits nothing on the population), so any future fitted preprocessing must use only
  training assets.
* **Replay exclusion** — replay assets never appear in the train, validation, or calibration
  partitions.

## Offline vs. incremental (training-serving) consistency

The single-asset incremental interface (`IncrementalFeatureState`) ingests observations one cycle
at a time and returns the current-cycle feature row, using the **same** `FeatureBuilder` code as
the offline batch path. Tests (`tests/unit/test_feature_consistency.py`) require the incremental
row to equal the offline row at every cycle, including state reconstruction from history after a
restart. The online path never requires future observations. Trajectories are short, so the state
keeps full history and the `adjust=True` EWM features remain exactly equal to the batch values.

## Output datasets

`make features` (or `uv run python scripts/build_features.py`) writes to
`data/features/cmapss/FD001/`:

| File | Content | Columns |
| --- | --- | --- |
| `train.parquet` | Train partition, model-ready | `asset_id, cycle, split, rul[, rul_capped], <features>` |
| `validation.parquet` | Validation partition | same as train |
| `calibration.parquet` | Calibration partition | same as train |
| `replay.parquet` | Replay partition | same as train |
| `test_features.parquet` | Official test features (all cycles) | `asset_id, cycle, split, <features>` (no targets) |
| `test_labels.parquet` | Official test RUL benchmark | `asset_id, final_cycle, rul` |
| `split_manifest.json` | The asset-level split assignment | — |
| `feature_manifest.json` | The feature definition and provenance | — |

Column groups are explicit: **identifiers** (`asset_id`, `cycle`), **metadata** (`split`),
**targets** (`rul`, optionally `rul_capped`; absent from test features), and **feature** columns.
Splits are kept in separate files; each file also carries a constant `split` column for
traceability.

### Feature manifest

`feature_manifest.json` lets a future training run identify exactly which feature definition and
split it used. It records: dataset name/subset, feature version, the Loop 2 processing-report
checksum, the split-manifest checksum, the feature configuration (families, windows, EWM spans,
`min_periods`, source columns), the ordered feature column names, the identifier/target/metadata
column groups, the RUL cap, the imputation policy (`null` = deferred), the inputs consumed (by
checksum), and per-output provenance (path, SHA-256, row and asset counts, null count, whether it
carries targets), plus row/asset counts by split, git commit, creation timestamp, and seed. The
`split_manifest.json` records the seed, strategy, and the asset IDs, counts, and row counts per
partition, linked to the source processing-report checksum.

## Behaviour guarantees

* **Input immutability** — the build only reads the Loop 2 processed layer and verifies each input
  Parquet against the processing-report checksum first; it never modifies the raw, validated, or
  processed layers.
* **Atomic writes** — outputs and manifests are written to a temp file and renamed.
* **Idempotency** — re-running verifies the recorded inputs, configuration, and output checksums;
  when everything matches nothing is rewritten (`already_built`).
* **Tamper detection** — a modified or missing output (or a changed Loop 2 input) fails with a
  clear error instead of silent repair; `--force` deliberately rebuilds.
* Generated feature data is **never committed to git** (`/data/` is gitignored).

## How to run

```bash
make acquire    # Loop 1: download + immutable raw layer + manifest
make process    # Loop 2: parse, validate, write Parquet + report
make features   # Loop 3: labels, splits, features, manifests
# equivalently: uv run python scripts/build_features.py [--data-dir DIR] [--seed N] [--rul-cap N] [--force]
```

## Known limitations

* **FD001 only.** The builder and splits are subset-agnostic; only the Loop 2 canonical count
  profile is FD001-specific.
* **Constant/near-constant sensors** (e.g. `operating_setting_3`, sensors 01/10/18/19) still get
  features generated (they degrade to constants/zeros). Feature **selection** by trend/information
  content is deliberately deferred to Loop 4 — Loop 3 generates the full, stable feature set.
* **Imputation and scaling are deferred to Loop 4.** Early-cycle rows contain nulls by design;
  downstream model pipelines must handle them (fit any imputation on training assets only).
* **Test↔RUL correspondence** is positional (row `i` ↔ asset `i + 1`), per the dataset readme, and
  cannot be cross-validated from file content alone.
